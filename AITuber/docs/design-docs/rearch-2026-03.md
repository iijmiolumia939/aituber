## 2. 現状アーキテクチャの課題分析

### 2-1. LipSync の二重駆動問題（最重要）

```
現状:
  VOICEVOX mora timing → VisemeEvent[] → BlendShape (母音 あいうえお + ARKit-18)
                               ↑ 並列に競合している
  A2F neural → ARKit-52 weights → BlendShape (ARKit-18 の口形状)

問題:
  - TTS viseme が発話の "音素タイミング" を制御
  - A2F neural が同じ blendshape インデックスに別の値を毎フレーム書き込む
  - 両者の補間が競合し、口の動きがぎこちなくなる可能性
  - globalStrength=2.5 という高倍率で誤魔化している

理想:
  一方に統一するか、明示的に A2F output を正とし TTS viseme を廃止
```

### 2-2. A2G DLL 未設置

```
現状:
  A2GPlugin.dll が存在しない → Audio2GestureController.IsReady = false
  SetEmotionGestureScale() のみ有効 (scale 0.4〜1.2 の placeholder)
  → 手/腕の動きが一切ない

問題:
  - NVIDIA A2G SDK はエンタープライズ向け (高コスト・大規模インストール)
  - SDK ライセンス要件が配布の障壁になりうる
  - PC スペック (RTX が必要) が高い
```

### 2-3. A2E の Python 側実行

```
現状:
  Python orchestrator で ONNX推論(CPU) → WS → Unity → BlendShape
  推論後に WS ラウンドトリップ (~数十ms) が発生

より良い案:
  Unity 側 com.unity.ai.inference (Sentis 2.5.0) が既にインストール済み
  → Unity 内部で GPU 推論 → WS メッセージ削減 → レイテンシ改善
```

### 2-4. VRM 形式の未活用

```
現状:
  QuQu U.fbx (Humanoid FBX) + UniVRM SpringBone のみ
  表情はすべて blendshape インデックスでハードコード

問題:
  - 表情の名前付き管理ができない (index マッピングが壊れやすい)
  - VRM 1.0 の Expression/Preset 機能が使えない
  - 他のキャラクターに差し替える際のコスト大

改善案:
  VRM 1.0 に完全移行 → UniVRM 1.0 Expression API を使う
  または VRMA (VRM Animation) を導入してモーションドリブン表現
```

## 4. リアーキテクチャ提案

### 提案 A: 漸進的改善（推奨・リスク低）

現行の WS JSON アーキテクチャを維持しつつ、コンポーネント単位で置換する。

```
Phase A-1: LipSync 統一化 (1〜2週)
  - A2F + TTS viseme → uLipSync ベースに統一
  - OR: A2F を正として TTS viseme を廃止
  - ResolveModelJsonPath() から uLipSync への切替は Audio2FaceLipSync.cs のみ
  - 効果: 口の動きの干渉解消、ビルド依存のDLL排除検討

Phase A-2: A2E → Unity Sentis 移行 (1週)
  - audio2emotion-v2.2/network.onnx を StreamingAssets に配置
  - Unity 内で com.unity.ai.inference で GPU 推論
  - send_a2e_emotion WS メッセージ廃止 → latency 削減 ~30ms
  - Python の audio2emotion.py は残すが WS 送信を削除

Phase A-3: AivisSpeech 対応 (3日)
  - TTSConfig に "aivisspeech" バックエンドを追加
  - VOICEVOX 互換 HTTP API なので tts.py の VoicevoxBackend を流用
  - より自然な日本語音声品質向上

Phase A-4: Procedural Body Gesture (2〜3週)
  - Animation Rigging パッケージで呼吸・体揺れ layered motion
  - 感情に応じた spine/shoulder の procedural 制御
  - A2G DLL なしで自然な体の動きを実現
```

### 提案 B: モダン化（中期・リスク中）

メッセージング層を近代化し、型安全性・レイテンシを改善する。

```
現状: Python WS server (port 31900) ← Unity WS client
      JSON text frames

改善: MessagePack binary frames (後方互換)
  - Python: msgpack-python で serialize
  - Unity: MessagePack-CSharp (MIT) で deserialize
  - WS 構造はそのまま維持
  - ペイロード 30〜50% 削減 → A2F chunk 転送効率向上

将来: gRPC (双方向 streaming)
  - A2F PCM チャンクも gRPC streaming で送信 (WS が不要に)
  -型安全スキーマ (proto) → AvatarMessage.cs 自動生成
  - Python: grpcio / Unity: grpc-dotnet
  - 導入コストが高いため将来オプション
```

### 提案 C: AI パイプライン強化（長期）

音声・感情・ジェスチャーを end-to-end で統合した次世代パイプライン。

```
現状: TTS(音声) → A2F(リップ) + A2E(感情) → Unity (別信号)

理想: Multimodal Audio → Face + Body を統合
  候補: NVIDIA ACE (Audio2Character Ensemble)
        = A2F + A2E + A2G を API で統合 (予定)
  または: SoundStorm / AudioPaLM ベースの直接アバター駆動

Inworld AI 統合:
  - キャラクター AI (LLM + TTS + Emotion + Gesture) を Inworld に委譲
  - Python orchestrator はポーリング + Safety のみに削減
  - ただし: キャラクター制御の柔軟性は低下・SaaS 依存
```

## 6. リスク・依存関係

| リスク | 影響 | 対策 |
|---|---|---|
| uLipSync が A2F 品質に届かない | LipSync 品質低下 | A2F との A/B 比較テスト実施 |
| Sentis (A2E) の ONNX モデル互換性 | A2E 動作しない | A2F ONNX も Sentis 変換して先行検証 |
| AivisSpeech の mora timing API | VisemeEvent 取得不可 | 既存 VOICEVOX を保持し fallback |
| Animation Rigging と A2G の競合 | ボーン回転の二重制御 | A2G 有効時は Rigging レイヤを無効化 |
| VRM 移行の表情マッピング崩れ | 誤表情・ゼロ表情 | EditMode テストで blendshape数 を検証 |

## 8. 現状維持の根拠（変えない部分）

以下は問題なく動作しており、変更コストが高い割にリターンが小さいため **現状維持**:

| 要素 | 理由 |
|---|---|
| WebSocket transport (port 31900) | 安定動作・再接続実装済み |
| Python Orchestrator 全体設計 | 725テスト passで品質担保済み |
| VOICEVOX + mora timing | 日本語品質申し分なし |
| Growth System (M1〜M8) | 独自価値・テスト充実 |
| Safety filter | NG/GRAY/OK 3段階で十分 |
| Contextual Bandit | ε自動調整・報酬モデル完備 |
| BehaviorSequenceRunner | behaviors.json ベースで拡張容易 |
